## 1. 基本方針

最初に次を固定する。

- 何を守るか
  個人情報、業務データ、秘密情報、管理者操作、他人データ参照
- 誰を防ぐか
  匿名利用者、一般利用者、同一 Workspace 内部者、スクリプト編集者、外部 API 側障害
- どの経路で侵入されるか
  Web App 直叩き、`google.script.run`、Execution API、トリガー、設定ミス、過剰 scope、ログ漏えい

確認項目:

- 最小権限原則を採用しているか
- 信頼境界を明示しているか
- データ境界を `ユーザー単位` `ドキュメント単位` `スクリプト全体` で分離できているか
- セキュリティ要件を機能要件と別に列挙しているか
- `誰が実行するか` と `誰のデータを触るか` が一致しているか

## 2. 実行面と権限モデル

### 2.1 Web App

確認項目:

- `appsscript.json` の `webapp.executeAs` を把握しているか
- `USER_DEPLOYING` / `USER_ACCESSING` の選択理由があるか
- `webapp.access` が `ANYONE` / `DOMAIN` / 組織限定のどれか明確か
- 所有者権限で実行する構成なら、利用者ごとの認可を独自実装しているか
- 状態変更 API を `GET` へ載せていないか
- 公開 URL を知るだけで実行できる構成になっていないか

危険信号:

- `executeAs` を理解しないまま `ANYONE` 公開
- UI 上は利用者単位でも、実処理は所有者権限で全データ参照
- 認可判断をメールアドレス取得可否へ依存

### 2.2 installable trigger / time-driven trigger

確認項目:

- trigger が「作成者権限で実行」されることを前提化しているか
- trigger が他ユーザー所有データや共有 Drive を横断しないか
- 同じ trigger の多重登録を防いでいるか
- retry や再実行時の冪等性を持つか
- lock や状態遷移により二重実行を抑止しているか

危険信号:

- 管理者が作った trigger で全社員データを一括処理
- 実行履歴やジョブ状態を持たず、途中失敗時に再送重複

### 2.3 Execution API / 外部呼び出し

確認項目:

- 呼び出し元を認証しているか
- 呼び出し元ごとの認可を持つか
- 実行結果やエラー詳細を外部へ出し過ぎていないか
- 実行対象関数を固定しているか
- 外部システム側の再試行で重複更新しないか

## 3. 入力・出力・Web エンドポイント

### 3.1 入力検証

確認項目:

- 型、長さ、文字種、列挙値、必須性を検証しているか
- ID を推測困難な形式へ制限しているか
- オブジェクトの追加フィールドを無視せず拒否または明示変換しているか
- URL、メール、日付、JSON を正規化しているか
- シート名、ファイル ID、プロパティキーへ未検証文字列を流していないか

### 3.2 出力制御

確認項目:

- クライアントへ stack trace、内部 ID、トークン断片を返していないか
- HTML を返すならエスケープを崩していないか
- JSON を返すなら契約外フィールドを混在させていないか
- ログ向け詳細と利用者向けメッセージを分離しているか

### 3.3 CSRF / 再送 / 乱用

確認項目:

- 状態変更操作を POST 以上へ限定しているか
- bearer token、署名付き nonce、アプリ固有トークンなど、実用的な呼び出し制御があるか
- 二重送信対策として冪等性キーを持つか
- 短時間大量実行、総当たり、資源枯渇へのレート制御を持つか

## 4. データ保存とテナント分離

### 4.1 PropertiesService

確認項目:

- `ScriptProperties` を全体共有領域として理解しているか
- `UserProperties` を利用者単位データに限定しているか
- `DocumentProperties` の共有範囲を理解しているか
- 機密情報を平文で置く必要性を説明できるか
- 永続データ破損時の復旧方針があるか

危険信号:

- ユーザー別データを `ScriptProperties` に混在保存
- API キーと利用者データを同じ JSON へ同居

### 4.2 Drive / Sheets / Docs 保存

確認項目:

- ACL 継承を理解しているか
- 共有ドライブや共有フォルダの閲覧者へ情報が漏れないか
- spreadsheet / document を秘密情報保管庫代わりにしていないか
- 行レベル分離が必要なのにシート 1 枚共有で運用していないか
- データ削除、訂正、保持期限を決めているか

### 4.3 CacheService

確認項目:

- キャッシュ不整合時のフォールバックを持つか
- キャッシュへ秘密情報や認可判断結果を長く残さないか
- キャッシュキーに利用者識別子や tenant 識別子を含めるか

## 5. 秘密情報と認証情報

確認項目:

- API キー、Webhook secret、OAuth client secret の保存場所を定義しているか
- `PropertiesService` 保存時、閲覧可能者を把握しているか
- 秘密情報のローテーション手順があるか
- ログ、エラー、通知へ秘密情報が出ないか
- テスト用認証情報と本番認証情報を分離しているか

推奨:

- 機密度が高い場合、Apps Script 単体へ閉じず、Secret Manager など外部秘匿基盤利用も検討
- トークンを長寿命固定値で埋め込まない
- 手動再設定が必要な箇所を runbook 化

## 6. 外部 API・外部通信

確認項目:

- `UrlFetchApp` 送信先ホストを allowlist 化しているか
- 利用者入力から完全な URL を受けてそのまま fetch していないか
- `http` を拒否し `https` を強制しているか
- request / response のタイムアウト、再試行、失敗時挙動を決めているか
- Webhook 検証署名、相手証明書、相互信頼前提を確認しているか
- 外部 API 応答を未検証のまま保存・画面反映していないか

危険信号:

- 任意 URL fetch → SSRF 的悪用
- 失敗時 full response をそのままログ出力
- 外部 API 障害で trigger が永久再実行

## 7. トリガー・バックグラウンド処理

確認項目:

- ジョブ投入条件、再実行条件、完了条件を状態機械として定義しているか
- `LockService` を共有更新前に取得しているか
- 冪等性キー、処理済み印、バージョン番号のいずれかを持つか
- 部分失敗時の補償処理があるか
- GAS quota 超過時の degradation を決めているか
- trigger 作成・削除を管理できるか
- 長時間処理を分割し、途中状態を安全保存しているか

危険信号:

- time-driven trigger を擬似キュー化しているが失敗時再送設計なし
- 手動起動関数と定期実行関数で認可前提が異なる
- 共有シート追記を lock なしで並列実行

## 8. 依存関係・マニフェスト・デプロイ

### 8.1 appsscript.json / OAuth scopes

確認項目:

- scope を必要最小限へ固定しているか
- 自動検出で不要な広い scope が混入していないか
- `gmail.modify` `drive` `spreadsheets` など強い権限に正当理由があるか
- Cloud project 連携設定を把握しているか

### 8.2 npm / bundling / clasp

確認項目:

- 依存ライブラリの保守状態と権限影響を見ているか
- build artifact に秘密情報を埋め込んでいないか
- `clasp push` 前に不要ファイル除外できているか
- ソースマップや debug 用定数を本番へ残していないか

### 8.3 権限管理

確認項目:

- スクリプト編集権限を最小人数へ絞っているか
- デプロイ権限者とレビュー責任者を分離しているか
- 本番デプロイ前承認フローを持つか

## 9. 可観測性・監査・インシデント対応

確認項目:

- 監査したいイベントを先に定義しているか
  認可失敗、権限昇格、秘密情報更新、データ大量取得、外部送信失敗
- ログから PII や秘密情報を除外しているか
- 失敗件数、再試行件数、quota 逼迫を監視できるか
- 異常検知後に trigger 停止、キー失効、デプロイ巻き戻しを実施できるか
- 保持期限と削除方針を決めているか

## 10. GAS 特有の危険信号

- `Session.getActiveUser().getEmail()` を認可の唯一根拠へ使う
- `executeAs` と `access` の意味を曖昧なまま公開
- `UserProperties` と `ScriptProperties` を混同
- `CacheService` 内容を信頼して認可判断
- installable trigger を管理者 1 名だけが作成
- 手動実行前提の初期化関数に強権限処理を混在
- spreadsheet を DB と秘密保管庫の両方へ使用
- 利用者入力 URL を `UrlFetchApp.fetch` へ直渡し
- エラー通知メールやログへ payload 全文出力
- `appsscript.json` scope を確認せずデプロイ
